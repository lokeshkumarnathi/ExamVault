Topic 1: Data Visualization Through Graph Representations

Topic 01
Definition: Graph representations are visual structures that use nodes (vertices) and edges (connections) to represent relationships, dependencies, and connectivity between data entities.

Core Components

Node (Vertex)

A single entity or data point in the graph. Represented as a circle or rectangle. Example: a person in a social network, a city on a map.

Edge (Link)

A connection between two nodes. Can be directed (with arrow) or undirected (plain line). Example: friendship between two people.

Weight

A numeric value assigned to an edge representing strength, distance, or cost. Example: road distance between cities.

Degree

Number of edges connected to a node. High degree = highly connected hub node. Example: a celebrity with many followers.

Types of Graphs

TypeDescriptionExample
Undirected GraphEdges have no direction — relationship is mutualFacebook friendship (A friends B = B friends A)
Directed Graph (Digraph)Edges have direction — relationship is one-wayTwitter follow (A follows B ≠ B follows A)
Weighted GraphEach edge has a numeric weight/costRoad map with distances, airline routes with prices
Unweighted GraphAll edges treated equallySimple friend network
Cyclic GraphContains cycles (loops back to start)Road network with roundabouts
Acyclic GraphNo cycles — one path between any two nodesFamily tree, file directory
Bipartite GraphNodes split into two groups; edges only between groupsMovie-actor relationships

Graph Layouts

  • Circular layout: Nodes placed on a circle. Good for showing all connections symmetrically.
  • Grid layout: Nodes placed in a regular grid. Good for geographic or matrix data.
  • Hierarchical layout: Nodes arranged in levels (top-down). Good for org charts, trees.
  • Force-directed layout: Nodes repel each other; edges attract connected nodes. Best for large complex networks.
  • Geographic layout: Nodes positioned using real-world coordinates (latitude/longitude). Maps.

Real-World Applications

  • Social networks: Facebook (friend graph), LinkedIn (professional connections), Instagram (follower/following)
  • Web graph: Web pages as nodes, hyperlinks as directed edges (basis of Google PageRank)
  • Transportation: Cities as nodes, roads/flights as weighted edges (GPS routing)
  • Recommendation systems: Users and items as nodes, ratings as edges (Netflix, Amazon)
  • Biological networks: Protein-protein interaction networks, gene regulatory networks

Topic 2: Graph-theoretic Graphics

Topic 02
Definition: Graph-theoretic graphics are visualizations rooted in graph theory — mathematical structures used to model pairwise relations between objects. They go beyond simple node-link diagrams to apply graph algorithms for layout, clustering, and analysis.

Force-Directed Layout

Force-Directed Layout: An algorithm that positions graph nodes by simulating physical forces — nodes repel each other like charged particles, while edges act like springs pulling connected nodes together. The system reaches equilibrium producing a natural, readable layout.
  • Nodes repel: F = k / d² (repulsive force inversely proportional to distance squared)
  • Edges attract: F = k × d (spring force proportional to displacement)
  • Iteratively computed until forces balance
  • Result: closely connected nodes cluster together, distant nodes spread apart
  • Used by: Gephi, D3.js forceSimulation, Cytoscape

Network Visualization

Network visualization represents complex systems of interconnected entities. Key metrics visualized include:

  • Centrality: Which nodes are most important? (PageRank, betweenness centrality)
  • Clustering: Which nodes form communities? (community detection)
  • Path length: Shortest path between nodes (Six degrees of separation)
  • Hub nodes: Highly connected nodes shown larger in visualization

Dependency Graphs

A directed graph showing dependencies between components. Widely used in software engineering:

Module Dependency Graph:
  [App.js]
     │
     ├──► [Router.js]
     │        │
     │        ├──► [Auth.js]
     │        └──► [Products.js]
     │
     └──► [Database.js]
              │
              └──► [Config.js]

(Arrows mean "depends on")

Topic 3: High-dimensional Data Visualization

Topic 03
Definition: High-dimensional data visualization is the process of representing data that has many attributes (dimensions) — often 5, 10, or even thousands — in 2D or 3D visual space where humans can perceive patterns.

The Curse of Dimensionality

Curse of Dimensionality: As the number of dimensions increases, data becomes increasingly sparse and distances between points become meaningless. Patterns that are clear in 2D become impossible to see in 100D.
  • In 1D: 10 data points cover space adequately
  • In 2D: need 100 points for same density
  • In 10D: need 10 billion points for same density
  • Visual impact: data appears uniformly spread — no clusters visible

Technique 1: Parallel Coordinates

Multiple parallel vertical axes, each representing one dimension. Each data item is drawn as a polyline crossing all axes at its value for each dimension.
Parallel Coordinates Plot (3 data records, 4 attributes):

Age  Income  Education  Health
 |      |        |         |
 |      |        |         |
50─────80K──────Masters────Good     ← Person 1
 |      |        |         |
25─────40K──────Bachelor───Fair     ← Person 2
 |      |        |         |
35─────60K──────PhD────────Excellent← Person 3
 |      |        |         |

Each line = one person; crossing lines = negative correlation

Clusters of parallel lines = similar records. Crossing lines between axes = negative correlation between those two dimensions.

Technique 2: Scatter Plot Matrix (SPLOM)

A grid of scatter plots where each cell shows a 2D scatter plot for every pair of dimensions. An n-dimension dataset produces an n×n matrix.
Scatter Plot Matrix (3 dimensions: X, Y, Z):

     X          Y          Z
  ┌──────┐  ┌──────┐  ┌──────┐
X │ diag │  │ X vs │  │ X vs │

Topic 4: Multivariate Data Glyphs — Principles & Practice

Topic 04
Definition: A glyph is a visual symbol or icon that encodes multiple data attributes simultaneously using visual properties such as size, shape, orientation, color, and texture. Each glyph represents one multivariate data record.

Why Glyphs?

When data has many attributes per record (e.g., a patient's age, blood pressure, cholesterol, BMI, glucose level), a simple bar chart can only show one attribute at a time. Glyphs encode ALL attributes into a single visual symbol, enabling comparison of entire profiles at a glance.

Types of Glyphs

1. Star Glyph (Radar / Spider Glyph)

Star Glyph (5 attributes):
          Speed
            |
         ___*___
        /   |   \
Memory-*----+----*-CPU
        \   |   /
         \  |  /
          \ | /
       RAM-*-*-Disk I/O
              |
           Graphics

Each axis = one attribute
Length from center = value (longer = higher)
Shape reveals the "profile" of the data record

2. Chernoff Faces

Chernoff Faces (1973): Multivariate data glyphs where each face feature encodes one data attribute. Introduced by Herman Chernoff. Humans are extremely good at recognizing and comparing faces, making this technique leverage natural cognitive ability.
Chernoff Face Attribute Mapping:
┌──────────────────────────────────┐
│  Feature         → Data Attribute│
│  Face shape      → Variable 1    │
│  Eye size        → Variable 2    │
│  Eye slope       → Variable 3    │
│  Nose length     → Variable 4    │
│  Mouth curvature → Variable 5    │
│  Mouth width     → Variable 6    │
│  Pupil position  → Variable 7    │
│  Eyebrow angle   → Variable 8    │
└──────────────────────────────────┘
Happy face = high values; Sad face = low values

Limitation: encoding is arbitrary — different feature-to-variable mappings give very different impressions. Features differ in perceptual salience (eyes more noticeable than ear size).

3. Other Glyph Types

Glyph TypeHow it worksBest for
Stick figure glyphBody part angles/lengths encode variablesMotion data, biomechanics
Color icon glyphColor channels (R,G,B) encode 3 attributesImage-like compact display
Tufteboxplot / sparklineMiniature chart embedded inlineTime-series in tables
Pie chart glyphSmall pie at each data point encodes compositionGeographic composition maps
Box glyphBox dimensions (width, height, color) = 3 variablesSimple 3-variable comparison

Principles of Glyph Design

  1. Perceptual accuracy: Use visual channels (position, size) that humans read most accurately
  2. Discriminability: Different data values must look perceptibly different
  3. Integral vs. separable: Choose encodings where attributes can be read independently
  4. Scalability: Glyph must remain readable at small sizes for large datasets
  5. Ordering: Place the most important attribute in the most salient visual channel

Applications

  • Healthcare: Patient health profile glyphs (blood pressure, BMI, cholesterol, glucose)
  • Financial analytics: Stock/portfolio glyphs showing price, volume, volatility, trend
  • Weather visualization: Wind rose glyphs showing speed, direction, and temperature
  • Sports analytics: Player performance glyphs showing speed, accuracy, endurance, passes, goals
⭐ Exam Point: Chernoff Faces are very frequently asked. Know: inventor (Herman Chernoff, 1973), the concept (facial features encode data variables), advantage (humans good at face recognition), and limitation (arbitrary mapping, varying salience of features).
Memory Trick: Glyph types = SCSPB — Star glyph, Chernoff faces, Stick figure, Pie glyph, Box glyph. "Stars and Chernoff See Pretty Boxes."

Topics 5 & 6: Linked Views & Linked Data Views

Topics 05 & 06
Definition: Linked Views (also called Coordinated Multiple Views or CMV) are interactive visualization systems where multiple views of the same dataset are simultaneously displayed and coordinated — a user action in one view (such as selecting data points) automatically updates all other views.

Why Linked Views?

No single visualization can show all aspects of complex data. A scatter plot shows correlations but not geographic distribution. A map shows locations but not statistical distributions. Linking multiple views allows analysts to see the same data from multiple perspectives simultaneously.

Core Mechanisms

1. Data Brushing

Data Brushing: The user selects (brushes) a subset of data points in one view by drawing a selection box or lasso. The selected points are immediately highlighted in all linked views.
Brushing Example:
View 1 (Scatter plot)     View 2 (Bar chart)     View 3 (Map)
    ·  · ·                 █ █ █ █               . . . .
   [·  · ·] ← brush       [█ █] ← highlighted   [. .] ← shown
    · ·  ·                  █ █ █                 . . .

Points selected in scatter → highlighted in bar → shown on map

2. Cross-Filtering

Cross-Filtering: Selecting a range or category in one view filters the data shown in all other views. Reducing the visible dataset in one chart automatically reduces it in all linked charts.
Cross-Filtering Example:
Filter applied: "Year = 2023" in timeline view

View 1 (Timeline): [2020][2021][2022][2023]
                                        ▲ selected
View 2 (Bar):      Shows only 2023 sales data
View 3 (Map):      Shows only 2023 geographic data
View 4 (Table):    Shows only 2023 records

3. Coordinated Multiple Views (CMV)

  • Multiple synchronized panels showing the same dataset differently
  • Interaction propagates through shared data model
  • Views can include: bar chart, line chart, scatter plot, map, table, heatmap
  • Real-time synchronization — no manual refresh needed

Linked Data Views Architecture

CMV Architecture:
                    ┌─────────────────┐
                    │  Shared Data    │
                    │     Model       │
                    │  (single source │
                    │   of truth)     │
                    └────────┬────────┘
              ┌──────────────┼──────────────┐
              │              │              │
         ┌────▼────┐   ┌─────▼───┐   ┌─────▼───┐
         │ View 1  │   │ View 2  │   │ View 3  │
         │ Scatter │   │  Map    │   │  Bar    │
         │  Plot   │   │         │   │  Chart  │
         └────┬────┘   └─────┬───┘   └─────┬───┘
              │              │              │
              └──────────────┴──────────────┘
                    Selection events
                    propagate to all views

Types of Linkage

Linkage TypeDescriptionExample
Full linkageAll interactions propagate to all viewsBrushing in one view highlights all others
Selective linkageOnly certain interactions are linkedZoom linked but color not linked
Directional linkageChanges flow in one direction onlyTimeline filters map but not vice versa

Real-World Examples

  • Tableau: Dashboard with bar chart + map + time series all linked by filter
  • Power BI: Cross-filtering between all visuals on a page
  • DC.js: Open-source library for linked D3.js charts with cross-filtering
  • Gapminder: Linked scatter plot + map + bar chart of world development data
  • Healthcare dashboard: Patient list + vital signs chart + medication history + lab results — all linked by patient selection

Topic 7: Visualizing Trees and Forests

Topic 07
Definition: A tree is a hierarchical data structure with a root node and child nodes arranged in parent-child relationships, with no cycles. A forest is a collection of disjoint trees. Tree visualization makes hierarchical relationships visually comprehensible.

Tree vs. Graph

FeatureTreeGraph
CyclesNo cycles allowedCycles allowed
HierarchyStrictly hierarchicalNot necessarily hierarchical
RootHas exactly one rootNo root concept
PathExactly one path between nodesMultiple paths possible
EdgesN nodes → N-1 edgesN nodes → variable edges
DirectionTop-down or child-parentAny direction

├── CTO
│   ├── Lead Engineer
│   │   ├── Frontend Dev

│   ├── Accounting

└── CMO
    ├── Marketing Team
    └── PR Team

Root = CEO; Leaf nodes = Individual contributors

Types of Tree Visualizations

1. Node-Link Diagram (Traditional Tree)

Classic tree drawn with nodes and connecting lines. Easy to understand. Scales poorly with very large trees (space wasted on white space).

2. Dendrogram

Dendrogram: A tree diagram used to show hierarchical clustering results. Horizontal lines connect clusters; the height of the connection shows the distance at which clusters merged. Used extensively in biology, genomics, and data mining.
Dendrogram (Hierarchical Clustering):
Height
  4 │         ┌───────────────┐
    │         │               │
  3 │    ┌────┴──┐         ┌──┴────┐
    │    │       │         │       │
  2 │  ┌─┴─┐  ┌─┴─┐     ┌─┴─┐  ┌─┴─┐
    │  │   │  │   │     │   │  │   │
  0 │  A   B  C   D     E   F  G   H
       └────────── Similar items cluster first ──────────┘

3. Treemap

Treemap: Hierarchical data displayed as nested rectangles. The area of each rectangle encodes a quantitative value (e.g., file size, market cap, population). Color can encode a second variable (e.g., change percentage).
Treemap (File System):
┌──────────────────────────────────────┐
│         /home directory              │
│ ┌────────────┐  ┌──────────────────┐ │
│ │  Documents │  │      Videos      │ │
│ │  (large)   │  │   (very large)   │ │
│ └────────────┘  └──────────────────┘ │
│ ┌──────┐  ┌─────────────────────────┐│
│ │Music │  │        Photos           ││
│ │(med) │  │      (medium-large)     ││
│ └──────┘  └─────────────────────────┘│
└──────────────────────────────────────┘
Area = file size. Larger area = larger folder/file.

4. Radial/Sunburst Tree

Hierarchical data displayed in concentric circles. Root at center; children expand outward. Excellent for showing proportions at each level of the hierarchy. Used in file system visualizers (WinDirStat, disk usage apps).

5. Icicle Plot (Flame Graph)

Hierarchical data displayed as horizontal bars stacked top-down (or bottom-up for flame graphs). Width represents size/value. Used in performance profiling to show call stack duration.

Visualizing Forests

A forest = multiple trees. Visualized by displaying trees side by side or as a combined network where forest-level relationships are shown.

Forest = Collection of independent trees:
Tree 1        Tree 2        Tree 3
  A             E             I
 / \           / \             \
B   C         F   G             J
   / \             \
  D   E             H

Applications of Tree Visualization

  • File systems: Folder hierarchy (Windows Explorer tree panel)
  • Organizational charts: Company reporting structure
  • Biological taxonomy: Species classification (Kingdom → Phylum → Class → Order)
  • Decision trees: Machine learning model explanation
  • XML/JSON structure: Data schema visualization
  • Genealogy: Family trees, ancestral charts
  • Network routing: Spanning tree of network topology
⭐ Exam Point: Treemap and Dendrogram are the two most frequently asked tree visualization types. Know Treemap: nested rectangles, area = value. Know Dendrogram: height = cluster distance, merges bottom-up.
Memory Trick: Tree viz types = NDTRI — Node-link, Dendrogram, Treemap, Radial/sunburst, Icicle/flame. "Nice Diagrams Tell Real Insights."

Exam Questions & Model Answers

2-Mark Questions

What is a node and edge in graph visualization?2 marks
A node (also called a vertex) is a fundamental unit of a graph representing an individual entity or data point. In a social network, each person is a node. An edge (also called a link or arc) is a connection between two nodes representing a relationship or interaction between the corresponding entities. In a social network, an edge between two person-nodes represents their friendship or connection. Edges can be directed (one-way, with an arrow) or undirected (two-way, plain line), and can carry weights (numerical values).
What is a Treemap?2 marks
A Treemap is a type of hierarchical data visualization that displays hierarchical data as nested rectangles. Each rectangle represents a node in the hierarchy, and the area of each rectangle is proportional to a quantitative value such as file size, revenue, or population. Color can encode a second variable such as percentage change or category. Treemaps were introduced by Ben Shneiderman in the early 1990s. They are highly space-efficient and widely used for visualizing file systems (disk usage tools), stock market portfolios, and website analytics by section.
Define Chernoff Faces and who invented them.2 marks
Chernoff Faces are a multivariate data glyph technique invented by statistician Herman Chernoff in 1973. Each face feature (such as eye size, mouth curvature, nose length, eyebrow angle, face shape) encodes one attribute of the data. The technique exploits the human brain's exceptional ability to recognize and compare faces — even subtle differences in facial features are immediately noticed. Each face represents one multivariate data record. Advantage: intuitive human perception. Limitation: the mapping of variables to face features is arbitrary and some features (eyes) are more visually salient than others (ear size), creating perceptual bias.
What is the Curse of Dimensionality?2 marks
The Curse of Dimensionality refers to the phenomenon where the volume of the data space increases so rapidly with the number of dimensions that the available data becomes extremely sparse. In low dimensions (1D, 2D), data points are dense and patterns are visible. As dimensions increase, data becomes sparse — distances between points become nearly equal, clusters disappear, and patterns become impossible to detect visually. For example, in 1D you need 10 points for adequate coverage; in 10D you need 10 billion. This is a fundamental challenge in high-dimensional data visualization, motivating dimensionality reduction techniques like PCA, t-SNE, and UMAP.
What is data brushing in linked views?2 marks
Data brushing is an interaction technique used in linked/coordinated multiple view (CMV) systems where the user selects (brushes) a subset of data points in one visualization view — typically by drawing a selection box or lasso around the points of interest. The selected data subset is then immediately highlighted or emphasized in all other linked views simultaneously. For example, in a dashboard with a scatter plot, bar chart, and geographic map all showing the same dataset, brushing a cluster of points in the scatter plot will highlight the corresponding bars in the bar chart and the corresponding locations on the map, enabling multi-perspective exploration of the same data subset.

5-Mark Questions

Explain parallel coordinates for high-dimensional data visualization with a diagram.5 marks
Introduction: Parallel coordinates is a technique for visualizing high-dimensional data by arranging multiple vertical axes side by side, each representing one data dimension. Each data record is represented as a polyline that passes through all axes at the record's value for that dimension.

Structure: Each vertical axis represents one attribute. Values are scaled along each axis. Each data item is drawn as a connected polyline crossing all axes at its corresponding values.

Diagram:
Age   Score  Study_hrs  Sleep_hrs
 |      |        |          |
22─────85─────────7──────────7   ← Student A
 |      |        |          |
20─────60─────────4──────────9   ← Student B
 |      |        |          |
24─────90─────────8──────────6   ← Student C
 |      |        |          |

Interpretation:
1. Lines staying parallel between two axes = positive correlation between those attributes
2. Lines crossing between two axes = negative correlation
3. A cluster of parallel, closely spaced lines = similar data records
4. An outlier line that diverges far from others = anomalous record

Advantages: Can handle 10–100+ dimensions simultaneously; reveals correlations; detects outliers.
Limitations: Crossing lines become cluttered with large datasets (overplotting); order of axes affects perceived patterns; difficult for untrained users.

Real-world use: Tableau uses parallel coordinates for multivariate exploration; used in bioinformatics for gene expression analysis.
Explain force-directed layout in network visualization.5 marks
Introduction: Force-directed layout is an algorithm for automatically positioning nodes in a network graph by simulating physical forces acting on the nodes. It produces visually appealing, meaningful layouts that naturally reveal clusters and community structure.

Physical Simulation:
1. Repulsive force: All pairs of nodes push each other apart. F_repel = k / d² (force inversely proportional to distance squared)
2. Attractive force: Edges act like springs, pulling connected nodes toward each other. F_attract = k × displacement
3. The algorithm iterates until all forces balance — the system reaches equilibrium.

Process:
Step 1: Place nodes randomly on canvas
Step 2: Calculate repulsive force between every pair of nodes
Step 3: Calculate attractive force for every edge
Step 4: Move nodes proportionally to net force
Step 5: Repeat until stable (low energy state)

Visual outcome: Densely connected nodes cluster together (communities visible). Loosely connected nodes spread apart. Hub nodes (high degree) naturally emerge at prominent positions.

Advantages: No manual positioning needed; reveals community structure automatically; aesthetically pleasing; handles undirected and directed graphs.

Limitations: Computationally expensive for large graphs (O(n²) per iteration); non-deterministic — different runs produce different layouts; may have poor convergence.

Tools: Gephi, D3.js forceSimulation, Cytoscape, NetworkX with spring_layout

10-Mark Essay Questions

Explain Coordinated Multiple Views (Linked Views) for visual data exploration with architecture, mechanisms, and examples.10 marks
Introduction: Coordinated Multiple Views (CMV), also known as Linked Views, is an advanced interactive visualization paradigm where multiple visualizations of the same dataset are simultaneously displayed and synchronized — any user interaction in one view automatically updates all other views. This enables analysts to explore data from multiple perspectives simultaneously, revealing insights impossible from any single view.

Motivation: No single chart type can show all dimensions of complex data simultaneously. A bar chart shows comparisons but not geographic distribution. A map shows location but not statistical trends. A scatter plot shows correlations but not time series. CMV solves this by linking all of these views together.

Architecture — Shared Data Model: All views connect to a single shared data model/repository. When a user interacts with any view, the interaction generates a selection event that is broadcast to all other views through the shared model. Each view then re-renders using only the selected/filtered subset. This ensures consistency across all views.

Core Mechanisms:

1. Data Brushing: The user draws a selection (brush) over data points in one view. The selected subset is highlighted in all linked views simultaneously. Example: brushing low-income points in a scatter plot highlights those same records in the map and bar chart.

2. Cross-Filtering: The user filters data in one view (e.g., selecting "Year = 2023" in a time-series view). All other views immediately show only data matching that filter. Unselected data dims rather than disappears. Example used by DC.js library extensively.

3. Zooming and Panning Linkage: Zooming into a time range in one view proportionally scales all time-based views. Example: zooming Jan–Mar in timeline also zooms all other time-encoded charts.

4. Hover Tooltips: Hovering over a data point in one view shows its details in a tooltip AND highlights the same record across all views.

Types of Linkage:
- Full linkage: All interactions propagate to all views
- Selective linkage: Only specific interactions are linked
- Directional linkage: Changes in View A affect View B but not vice versa

Architecture Diagram:
         ┌─────────────────────┐
         │    Shared Data      │
         │       Model         │
         └──────────┬──────────┘
     ┌──────────────┼──────────────┐
     │              │              │
┌────▼────┐   ┌─────▼───┐   ┌─────▼───┐
│Scatter  │   │  Map    │   │  Bar    │
│  Plot   │   │         │   │  Chart  │
└────┬────┘   └─────┬───┘   └─────┬───┘
     └──────────────┴──────────────┘
          Selection events propagate

Real-World Examples:
1. Tableau: Full CMV with cross-filtering across all dashboard visuals
2. Microsoft Power BI: Click any bar → all other visuals filter by that dimension
3. DC.js / Crossfilter.js: Open-source library built specifically for linked chart cross-filtering
4. Gapminder: Linked scatter plot + map + bar chart + time-series of world development data
5. Healthcare dashboard: Patient list → vital signs chart → medication history → lab results — all linked by patient selection
6. Finance: Stock screener with linked price chart, volume histogram, and sector map

Advantages:
- Multiple perspectives simultaneously; faster insight discovery
- Supports complex multi-variable hypothesis testing
- Reduces cognitive load of switching between charts
- Natural workflow: select → filter → explore

Disadvantages:
- Complex to implement; requires sophisticated software
- Performance issues with very large datasets
- Cognitive overload if too many views are linked
- Requires digital medium — not suitable for print

Conclusion: CMV represents the most powerful paradigm for exploratory data analysis. By coordinating multiple complementary views, analysts can simultaneously see correlation, distribution, geography, and temporal patterns — transforming data from numbers to narrative.
Explain tree visualization techniques — node-link diagrams, treemaps, dendrograms, and sunburst charts — with examples and applications.10 marks
Introduction: Tree visualization is the graphical representation of hierarchical data structures — data organized in parent-child relationships with a single root. Hierarchical data appears everywhere: file systems, taxonomies, organization charts, decision trees, and evolutionary biology. Different visualization techniques suit different types of hierarchical data and analysis tasks.

Definition of Tree: A tree is a connected, acyclic graph with a designated root node. N nodes have exactly N-1 edges. There is exactly one path between any two nodes. A forest is a collection of disjoint trees.

1. Node-Link Diagram:
The classic representation where nodes are drawn as circles/boxes and parent-child relationships as connecting lines. Intuitive and easy to understand. Best for small-to-medium trees where white space is not a concern. Used in: org charts, family trees, decision trees, XML/JSON visualization. Limitation: wastes space; large trees become unreadable.

2. Treemap:
Introduced by Ben Shneiderman (1990s). Hierarchical data displayed as nested rectangles. The area of each rectangle is proportional to a quantitative attribute (file size, revenue). Color encodes a second attribute (growth rate, category). Space-efficient — uses 100% of display area. Best for: file system visualization (WinDirStat), stock portfolio analysis (heatmaps showing market cap), website traffic by section. Limitation: difficult to show deep hierarchies; hard to compare non-adjacent items.

Diagram:
┌──────────────────────────────────┐
│         /home directory          │
│ ┌──────────────┐  ┌────────────┐ │
│ │ Videos (40%) │  │Photos(30%) │ │
│ ├──────┬───────┘  └────────────┤ │
│ │Docs  │    Music (10%)       │ │
│ │(20%) │                      │ │
│ └──────┴──────────────────────┘ │
└──────────────────────────────────┘
Area = file size; Color = file type

3. Dendrogram:
A tree diagram used to show results of hierarchical clustering. Items are arranged at the leaves; branches merge bottom-up as similar items are grouped. The height at which two branches merge indicates the distance/dissimilarity between those clusters. Used in: biology (evolutionary trees/phylogenetics), bioinformatics (gene expression clustering), data mining (customer segmentation), document clustering. Limitation: hard to read with many items; branch height must be carefully labeled.

Diagram:
Distance
    4│          ┌──────────────────┐
     │          │                  │
    3│     ┌────┴───┐          ┌───┴───┐
     │     │        │          │       │
    2│  ┌──┴─┐    ┌─┴──┐    ┌──┴─┐  ┌─┴─┐
     │  │    │    │    │    │    │  │   │
    0│  A    B    C    D    E    F  G   H

4. Radial/Sunburst Tree:
Hierarchical data displayed in concentric rings. Root node at center; children expand outward in rings. Each ring represents one level of the hierarchy. Angle of each arc represents the relative proportion of that node. Aesthetically pleasing; good for showing proportions at multiple hierarchical levels simultaneously. Used in: file system visualization (disk usage apps), taxonomic classification, website section analysis. Limitation: outer rings get crowded with many small items.

5. Icicle Plot / Flame Graph:
Hierarchical data as horizontal bars stacked top-down (icicle) or bottom-up (flame). Width represents quantitative value. Used in performance profiling to show call stack execution time — the bottom bar (root function) contains all time; wider child bars = more CPU time spent.

Forest Visualization: A forest (collection of disjoint trees) is visualized either by placing trees side-by-side or by introducing a virtual root node connecting all tree roots.

Applications:
1. File systems: Windows Explorer folder tree; WinDirStat treemap
2. Organization charts: Corporate hierarchy node-link diagrams
3. Biology: Phylogenetic trees (evolutionary relationships), dendrograms
4. Machine learning: Decision tree visualization (model explanation)
5. Web navigation: Breadcrumb trails, site maps as trees
6. Data mining: Hierarchical clustering dendrogram

Conclusion: Tree visualization encompasses a rich set of techniques — from the intuitive node-link diagram for small hierarchies to the space-efficient treemap for large ones. Choosing the right technique depends on the size of the hierarchy, whether quantitative values need encoding, and whether the primary task is exploring structure or comparing sizes.
